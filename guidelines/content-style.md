# Content style guide

Formatting, grammar, and structural conventions for all Red Door Collaborative written output.

> **Status:** Working document. Rules here are current practice — add to this file when new decisions are made. Reference `voice-tone.md` for personality and register guidance.

---

## Page and document structure

### Headers
- Use sentence case — never title case
- H1: page title only (one per page)
- H2: major sections
- H3: subsections within a section
- Don't skip heading levels (H1 → H3 with no H2)

### Paragraphs
- One idea per paragraph
- 2–4 sentences as a target; break anything longer
- White space between paragraphs is not wasted space — it aids comprehension

### Lists
- Use bulleted lists for unordered items (3 or more)
- Use numbered lists for sequences and steps
- Parallel structure within a list (all items start the same way grammatically)
- No period after bullet items that are not complete sentences; period after complete sentences

### Tables
- Use for comparative or structured data — not as a design element
- Every column header should be meaningful
- Align numbers right; align text left

---

## Formatting

### Bold
Use for genuinely critical information, key terms on first use, or UI element names. Do not bold for decorative emphasis.

### Italic
Use for titles (of books, films, reports), foreign phrases, or subtle stress. Do not use for scare quotes.

### Code and technical strings
Use `inline code` for:
- File names, paths, and extensions
- CSS class names and tokens
- Short code snippets inline in prose
- URLs when shown as literal text (not as links)

Use code blocks for multi-line code or anything meant to be copied.

### All caps
Use only for: eyebrow labels (via the `.eyebrow` CSS class), acronyms, and keyboard shortcuts. Not for emphasis in body copy.

## Dates and times

- Dates: June 5, 2026 — not 06/05/26 or 6-5-26 in prose
- Fiscal year: FY26, not FY2026 or fiscal year 2026 (in running text)
- Time: 9 AM, not 9:00 AM or 9am (use space before AM/PM)
- Ranges: June 1–5, 2026 (en dash, no spaces)

## Numbers

| Context | Rule |
|---------|------|
| One through nine | Spell out |
| 10 and above | Numerals |
| Starting a sentence | Always spell out |
| Percentages | Always numerals (5%, not five percent) |
| Large numbers in prose | 1.2M, 48K |
| Currency | $1.2M, $48K — no trailing zeros for whole amounts |

---

## Capitalization

| Item | Rule |
|------|------|
| Job titles (general) | Lowercase: "she is a project manager" |
| Job titles (before name) | Capitalize: "Head of People Brian Jensen" |
| Team and department names | Capitalize: the Creative team, the Operations team |
| Product and service names | Follow the owner's capitalization (Azure, SharePoint) |
| Headings | Sentence case always |
| Button labels | Sentence case always |

---

## Acronyms and abbreviations

- Spell out on first use, then abbreviate: "Customer Success Marketing (CSM)"
- For widely known acronyms (AI, HR, ROI), no need to spell out
- Don't create new internal acronyms without documenting them

---

## Links

- Link text should describe the destination: "view the brand guidelines" — not "click here"
- Don't use bare URLs in prose unless showing the URL is the point
- External links: open in new tab (`target="_blank"`) for client-facing pages
- Internal links: same tab

---

## File naming (markdown and content files)

Follow the convention in `rdc-knowledge-base`:
- Repo files: `kebab-case.md` (lowercase, hyphens)
- SharePoint: `CLIENT-FY26-PO#-Descriptor-Rev.ext`
